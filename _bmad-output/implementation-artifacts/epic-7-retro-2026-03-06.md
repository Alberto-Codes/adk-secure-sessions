## What Went Well

### Spike-gated architecture pivot eliminated risk before commitment

Story 7.1 was a GO/NO-GO gate. 8/8 prototype tests passed. The key discovery -- ADK V1's `_merge_state()` uses Python-side `dict | delta`, not SQL `json_patch` -- unlocked the entire epic. The team had evidence before committing to rewriting 800 lines of production code.

### Massive complexity reduction

~800 lines of raw aiosqlite SQL replaced by ~130 lines of DatabaseSessionService wrapper. Zero CRUD method overrides. The `EncryptedJSON` TypeDecorator handles all encryption at the ORM boundary. Less code to maintain, less to break, less to understand.

### Story-to-story intelligence chain

Every story had a "Previous Story Intelligence" section referencing the prior story's learnings. 7.1 findings -> 7.2 ADR -> 7.3 implementation -> 7.4 tests -> 7.5 docs -> 7.6 scope. No context was lost between stories.

### TEA-driven cross-cutting test pipeline

Six stories, six genuine test improvements -- all sourced from the Coverage Gap Document (`test-review.md`). T036 deeply nested dict, T037 tampered envelope headers, concurrent event appends, tampered ciphertext-at-rest, 768-line test file split into 3 focused files, shared fixture extraction. The TEA pipeline (from Epic 6 retro action item) worked consistently.

### Code reviews caught real issues every time

7.1: 3 MEDIUM (sync/async test honesty, constant imports, file list). 7.2: 3 MEDIUM + 3 LOW (claim precision, missing AC5 content in Consequences). Every review found genuine problems, even for docs-only stories.

### Multi-database support for free

SQLite, PostgreSQL, MySQL, MariaDB -- all via DatabaseSessionService's dialect handling. No additional code needed. This superseded Epic 4 Stories 4.1-4.3, eliminating future work rather than deferring it.

## What Didn't Go Well

### Spike-to-production gap

The spike validated the happy path. Production implementation (Story 7.3) discovered edge cases the spike couldn't:
- `DontWrapMixin` needed on exception classes -- SQLAlchemy wraps TypeDecorator exceptions in `StatementError`, breaking the library's error contract
- Unencrypted data detection was non-trivial -- plaintext JSON partially decodes as valid base64
- `_SchemaClasses` duck-typing needed careful production handling (spike used fragile `__new__` bypass)

### Workflow config updates carried across two retros

Epic 6 retro committed to formalizing markdown rendering validation and stale-claim audit in workflow configs. Both were done manually in practice during Epic 7 but never formalized in workflow YAML files. This is the second consecutive retro flagging this.

### Internal method coupling accepted as conscious risk

`_get_schema_classes()` and `_prepare_tables()` are underscore-prefixed ADK internals. Sentinel tests mitigate the risk, but we're coupling to undocumented API. Epic 3 doubles down on this coupling with a second backend.

---

## Key Insights

1. **Spikes de-risk the happy path; production finds the edge cases** -- DontWrapMixin, unencrypted data detection, exception propagation all emerged post-spike. Budget for edge-case discovery in the production story.
2. **Process improvements backed by artifacts stick; memory-based ones don't** -- TEA pipeline succeeded because `test-review.md` is a concrete artifact. Workflow config updates failed because they relied on someone remembering.
3. **Evidence -> commitment -> implementation** -- The sequence spike (7.1) -> ADR (7.2) -> rewrite (7.3) -> tests (7.4) -> docs (7.5) -> scope (7.6) was the right order. Each step built on proven ground.
4. **Complexity reduction is a feature** -- 800 -> 130 lines. The best code is code you delete.
